# ADR-0002: 单维护者阶段保留 PR 和 CI，取消独立 GitHub 审批要求

- 状态：Accepted
- 日期：2026-07-20
- 决定者与批准者：项目负责人（明确指令：“可以，按照你推荐的来做吧，注意更新文档告知做了这个操作”）
- 生效日期：2026-07-20

## 问题

`main` 已启用 PR、`L0 governance check`、管理员强制、禁止强推和禁止删除；其中“至少 1 项审批”在当前只有一个 GitHub 账号的情况下无法满足。GitHub 不允许 PR 作者使用同一账号审批自己的 PR，导致已获项目负责人接受的 P0 仍不能合并。

## 候选方案

1. 解除整个 `main` 保护：可合并但会失去 PR、CI、管理员强制和推送安全门禁，不可接受。
2. 新增独立的具写权限审批账号：保留 1 项审批，但当前没有该账号，增加维护成本。
3. 保留 PR、状态检查、管理员强制、禁止强推和删除，取消独立审批数量要求：适配单维护者，同时保留可审计变更路径和自动化门禁。

## 决定

选择方案 3。`main` 必须继续通过 PR 合并，且 `L0 governance check` 必须成功；管理员不得绕过规则，强推和删除继续禁止。将独立审批数从 1 调整为 0。项目负责人在聊天中的明确阶段接受、Project Lead 阶段报告和独立 Test Agent 验收报告共同构成单维护者的人工验收证据。

## 影响

- 单账号可以在 PR 和成功 CI 后合并自己的变更，无法获得 GitHub 内的第二人审批。
- 不允许直接推送、失败 CI 合并、强推或删除 `main`；这些远端门禁不得因本决定被移除。
- 新增任何拥有写权限的维护者后，必须复审本 ADR，并恢复至少 1 项独立审批。
- P0 阶段报告必须记录本调整、远端保护复验结果和合并证据。

## 验证与复审

1. GitHub 保护 API 显示审批数量为 0，同时 PR、`L0 governance check`、管理员强制、禁止强推和删除仍为启用。
2. PR #1 在 CI 成功后不再因审批阻塞，并仅通过受保护的 PR 路径合并。
3. 首次新增协作者、P1 启动或任意安全/数据范围变化前复审。
