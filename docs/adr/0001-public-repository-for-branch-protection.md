# ADR-0001: 将 P0 GitHub 仓库改为公开以获得 main 分支保护

- 状态：Accepted
- 日期：2026-07-20
- 决定者与批准者：项目负责人（明确指令：“没关系，转公开吧”）
- 生效日期：2026-07-20

## 问题

P0 要求私有 GitHub 单仓库和受保护的 `main`。在 `G31ght/AskLily` 私有仓库上配置 GitHub 分支保护时，GitHub REST API 返回 HTTP 403，要求升级 GitHub Pro 或将仓库公开。没有远端保护就无法证明禁止直接推送、PR 门禁和必需状态检查已强制执行。

## 候选方案

1. 保持私有并依赖文档和人工约束：不满足“受保护 main”，不能关闭 P0。
2. 迁移到支持私有分支保护的 GitHub 计划或组织：保留私有性，但当前不可立即验证，需额外账户或组织决策。
3. 将仓库公开后启用 `main` 保护：可立即取得 P0 所需的远端强制门禁，但仓库内容对公众可见。

## 决定

选择方案 3。项目负责人于 2026-07-20 明确批准公开仓库；Project Lead 将仓库公开并立即配置 `main` 的 PR、审批和 `L0 governance check` 门禁。

## 影响

- 代码、文档和 Git 历史可被公开访问；此决定不允许提交真实数据、密钥、私有基础设施信息或任何未批准的业务实现。
- P0 当前候选经 `git ls-files` 和敏感模式扫描复核，仅包含目录占位符、治理文档、模板和离线 CI；未发现业务源文件、真实数据或密钥。
- `.gitignore` 与远端审查仍是防线，但不替代提交前的人工敏感信息审查。
- P0 阶段报告和 PR 说明必须记录此基线变化；未来若需要私有数据或实现，须重新作出 ADR 并迁移到支持私有分支保护的托管环境。

## 验证与复审

1. GitHub 仓库可见性显示为 `PUBLIC`。
2. `main` 分支保护 API 显示 PR、至少一项审批、`L0 governance check`、管理员强制、禁止强推和删除已启用。
3. 草稿 PR #1 的远端 CI 通过；其不得绕过保护直接合并。
4. 在首次真实数据、密钥、私有化部署或 P1 业务实现启动前复审本 ADR。
